### 오픈마켓 서비스

오픈마켓 서비스는 **판매자**와 **구매자**를 구별하여 판매자가 상품을 등록, 판매하며 구매자는 구매하는 서비스입니다.

## 웹 컴포넌트 가이드

이 프로젝트는 Web Components를 사용하여 재사용 가능한 UI 요소를 만듭니다.

### 자동 컴포넌트 등록 시스템

`vite-plugin-auto-components.js` 플러그인이 HTML 파일에서 커스텀 태그를 자동으로 감지하고 해당 컴포넌트를 임포트합니다.

#### 태그 형식 규칙

- **형식**: `<folder-file>` (kebab-case)
- **예시**:
  - `<hodu-footer>` → `src/component/hodu/footer.js`
  - `<imput-button>` → `src/component/imput/button.js`
  - `<user-profile-card>` → `src/component/user/profile-card.js`

## 에러와 에러 해결

### 강우석

Checkbox 컴포넌트 문제 발생

발생상황
체크박스 웹 컴포넌트 구현 중 이슈 발생

원인
컴포넌트 정의 및 스타일링 문제

해결
체크박스 컴포넌트 재작성 및 CSS 연결 수정

---

컴포넌트화 스타일 오류

발생상황
웹 컴포넌트로 전환 후 스타일이 적용되지 않음

원인
Shadow DOM 내부에서 외부 CSS 파일 접근 불가

해결
index.html에서 불필요한 코드 59줄 제거, input.js 컴포넌트 내 스타일 로드 방식 수정

---

상위 컴포넌트에서 하위 JS 파일 불러오기 실패

발생상황
부모 컴포넌트에서 자식 컴포넌트 JS 파일을 import 할 수 없음

원인
컴포넌트 간 의존성 및 경로 문제

해결
vite-plugin-auto-components.js 파일을 99줄 추가하여 자동 컴포넌트 로드 기능 구현, footer.js, face.js, instar.js, youtube.js 등 경로 수정

---

회원가입 전화번호 중복 검사 오류

발생상황
회원가입 페이지에서 전화번호 유효성 검사 중 중복 검사가 정상적으로 작동하지 않음. 이미 등록된 전화번호로 가입 시도 시 에러 처리가 되지 않거나, 반대로 사용 가능한 번호임에도 중복으로 판단되는 현상 발생. API 응답은 정상이나 클라이언트에서 응답 처리 로직에 문제 발생.

원인
API 응답 데이터 구조를 정확히 파악하지 못해 잘못된 속성에 접근. 비동기 처리 타이밍 이슈로 검사 결과가 반영되기 전에 다음 단계로 진행. 전화번호 형식 변환 과정에서 하이픈 유무 등 포맷 불일치.

해결
API 응답 구조를 정확히 파악하여 올바른 속성(data.Success) 사용. 비동기 함수는 await로 결과를 기다린 후 다음 로직 실행. 데이터 포맷 통일(전화번호 하이픈 처리)은 클라이언트에서 미리 처리. try-catch로 네트워크 오류 등 예외 상황 대비.

결제 페이지 바로구매 무시 문제

발생상황
상품 상세 페이지에서 바로구매 버튼 클릭 시 결제 페이지로 이동하지만, 실제로는 장바구니에 담긴 상품들이 표시됨. 바로구매한 상품 정보가 무시되고 기존 장바구니 데이터가 우선 적용되는 현상. 사용자가 의도하지 않은 상품으로 결제 진행될 위험.

원인
결제 페이지 로딩 시 무조건 장바구니 API를 호출하여 데이터를 가져오는 로직. 바로구매와 장바구니 결제를 구분하는 플래그나 파라미터가 없음. sessionStorage로 전달한 바로구매 정보를 읽지 않거나, 장바구니 데이터가 덮어씌우는 문제.

해결
페이지 간 데이터 전달 시 명확한 키 구분(orderProduct vs orderItems). 바로구매 데이터를 먼저 확인하여 우선순위 부여. sessionStorage는 일회성 데이터 전달에 적합하며, 사용 후 removeItem()으로 삭제하여 충돌 방지. 주문 API 호출 시에도 order_type으로 바로구매(direct_order)와 장바구니(cart_order)를 명확히 구분. 단일 상품도 배열로 감싸서 처리하면 렌더링 로직을 통일할 수 있음.

### 김남희

커스텀 컴포넌트가 렌더링되지 않거나 이벤트가 동작하지 않음

발생상황
logo-component, button-medium, input-id 등 커스텀 엘리먼트를 적용했으나 화면에 표시되지 않거나, 화면에는 보이더라도 클릭/submit 등의 이벤트가 정상적으로 동작하지 않는 문제가 발생하였습니다.

원인
컴포넌트 정의 스크립트가 로드되지 않아(경로 오류, 모듈 로드 실패 등) 커스텀 엘리먼트가 등록되지 않는 경우가 있었습니다. innerHTML 기반으로 DOM을 생성하는 구조에서 렌더링 이전에 querySelector를 수행하면 요소 탐색에 실패하여 이벤트가 바인딩되지 않습니다. 이벤트 바인딩 로직이 여러 위치에 분산되면 일부 이벤트가 누락되거나, 반대로 중복 등록될 수 있습니다.

해결
컴포넌트 스크립트 import 경로 및 type="module" 설정을 점검하여, 페이지 진입 시 컴포넌트 정의가 반드시 실행되도록 정리하였습니다. Web Component 생명주기에서 connectedCallback() 호출 시 render()를 먼저 수행한 뒤, setupEventListeners()에서 this.querySelector(...)로 DOM을 탐색하여 이벤트를 등록하도록 패턴을 고정하였습니다. 이벤트 등록은 가능한 한 setupEventListeners() 한 곳에서 일괄 관리하여 누락/중복을 예방하였습니다.

로그인 입력값 조회 실패 및 입력 검증 UX 미흡

발생상황
로그인 폼에서 아이디/비밀번호를 입력했음에도 값이 정상적으로 읽히지 않거나 null로 처리되는 문제가 발생하였고, 입력 누락/불일치 상황에서 사용자 경험이 충분히 매끄럽지 않았습니다(어떤 입력칸을 다시 입력해야 하는지 불명확, 재입력 동선 비효율).

원인
input-id 등 커스텀 입력 컴포넌트 내부에 실제 input이 존재하는 구조인 경우, 외부에서 FormData 또는 단순 selector로 입력값을 조회하면 실패할 수 있습니다. name 속성이 실제 input에 반영되지 않거나 누락되면 querySelector('input[name="..."]') 방식이 동작하지 않습니다. 검증 로직이 메시지 표시에만 머무르면 사용자는 다음 행동(어느 칸을 고쳐야 하는지)을 즉시 수행하기 어렵습니다.

해결
폼 내부에서 진짜 input을 name 기준으로 직접 탐색하는 방식으로 입력값 조회 방식을 통일하였습니다. 입력칸 탐색 실패 시 즉시 검증 메시지 및 콘솔 경고를 출력하여 문제 원인을 빠르게 확인할 수 있도록 하였습니다. 입력 누락/불일치 상황에서 아이디 누락 시 아이디 input 초기화 후 포커스 이동, 비밀번호 누락 시 비밀번호 input 초기화 후 포커스 이동, 로그인 실패(불일치) 시 비밀번호만 초기화 후 포커스 이동하는 UX 정책을 적용하여 사용자의 재입력 동선을 단순화하였습니다.

## 개발하면서 느낀점

### 강우석
> 팀장으로서 온라인 인원과 합을 맞추는것이 처음이었는데 디스코드와 시간에 크게 제약을 받지않는 업무 위주로 배분하여 줌 으로서 업무일정에 큰 차질이 없게 하려고 생각을 많이 했던거같고 , 다들 개발 실력이 제각각이다 보니 다른 사람과 업무 조율하는 파트에서 우선 업무를 평등하게 주고난 후 팀원의 해결방식과 해결속도에 맞춰 업무 배분량을 조절하는 방식으로 하는 방식을 생각하게 되었습니다.

### 강주현
> 컴포넌트 기반 아키텍처를 처음 도입하며 설계와 재사용 구조 구축에 시행착오를 겪었지만, 프로젝트가 진행되며 같은 UI 요소를 여러 페이지에서 일관되게 사용할 수 있다는 장점을 체감했다. 팀원들과 각자의 관점에서 문제 해결 방법을 논의하고 코드 리뷰를 주고받으며 협업의 가치를 배웠다. 매일 작업 내역과 이슈를 공유하며 팀 전체의 진행 상황을 파악했고, Git 브랜치 전략, Pull Request, 코드 리뷰 등 실무 협업 방식을 자연스럽게 익혔다. 회원가입, 장바구니, 결제 시스템 등이 하나씩 완성되며 큰 성취감을 느꼈고, 다음 프로젝트에서는 컴포넌트 설계 단계부터 재사용성과 확장성을 고려하며 코드 리뷰 문화를 적극 활용해 코드 품질을 높이고 싶다.

### 김남희
> 개발 과정에서의 어려움은 문제를 대하는 태도와 코드 작성·수정 방식에서 비롯된다는 점을 인식하게 되었다. 기본 규칙을 필요 이상으로 엄격하고 무겁게 받아들이고, 매번 완전히 이해해야만 하는 대상으로 느꼈다. 특히 여러 요소가 유기적으로 연결된 상황에서는 사소한 수정도 큰 오류로 이어질 수 있다는 부담을 크게 느꼈다. 그 과정에서 팀원들의 도움을 통해 문제를 함께 해결하며 많은 것을 배울 수 있었고, 협업의 중요성과 장점을 다시 한번 느끼는 계기가 되었다.


### 김서준
> 1차 프로젝트를 개인으로 진행할 때는 해결하기 어려운 문제에 부딪히면 오로지 인터넷 검색에 의존하거나 혼자 끙끙 앓으며 해결해야 했기에, 막막함과 심리적 부담감이 커서 힘든 순간이 많았었다. 하지만 이번 팀 프로젝트를 수행하면서는 나보다 앞서 나가는 팀원에게 조언을 구해 효율적으로 문제를 해결할 수 있었고, 모두가 답을 찾기 어려운 상황에서는 함께 머리를 맞대고 고민하며 최선의 방향을 찾아 나아 갈 수 있었다. 이런 집단지성의 과정이 서로에게 큰 의지가 되었음은 물론, 혼자였다면 포기했을 상황에서도 다시 나아갈 수 있는 원동력이 되어 프로젝트 완수에 큰 도움이 되었다고 생각한다.
